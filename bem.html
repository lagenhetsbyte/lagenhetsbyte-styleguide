BEM

In large, complex and long-term projects, organizing your code is the key to efficiency. No matter which methodology you choose, you will get advantages of structured CSS.

Lägenhetsbyte.se is using BEM; it's clear, provides a good architecture and has recognizable terminology. BEM is the abbreviation of the key elements of the methodology — Block, Element and Modifier.

  • Block → stand-alone entity, and is meaningful on its own.
  • Element → is part of a block, always tied to the block.
  • Modifier → changes appearance or behavior on a block or element.

Example

Let's say we want to display some user data. We'll need to create a user component; a block. In this block we'll print a name and a description. These will be elements, always children of the user component. Also, there will be a list of user data. This kind of list also appears elsewhere, it's not unique for this particular user component. So it's a block. In the list, there are some list items (elements). One of these items is special: maybe it has a certain color, or a different font size. That would count as a modifier. 

<div class="user">
    <h1 class="user__name"></h1>
    <p class="user__description"></p>
    <ul class="list">
        <li class="list__list-item"></li>
        <li class="list__list-item"></li>
        <li class="list__list-item list__list-item--special"></li>
    </ul>
</div>

Now, let's assume we must (for whatever the reason) wrap the user name in a div. What shall we call it? .user__name--container? No, instead try to think of each block as a mini-DOM in its own right, with a header, a body, and a footer: 

...
<div class="user__header">
    <h1 class="user__name"></h1>
</div>
...

Please note:

  • The element .user__name is first and foremost a child of .user, not .user__header. Do not name it .user__header__name.
  • The .user__description and the .list could – if there is a need for it – be wrapped in .user__body.
